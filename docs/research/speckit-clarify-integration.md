# Research: `speckit.clarify` Integration Contract for 012 Review-Spec

**Date**: 2026-04-11
**Purpose**: Input for `012-review-model` plan. The 012 brainstorm
assumed `speckit.clarify` handles the sharpening layer so `review-spec`
can focus on adversarial concerns. This document captures the exact
output shape and scope boundaries so 012's plan can define a precise
integration contract.
**Source**: Session audit of `github/spec-kit` repomix pack. The
original research subagent returned a detailed breakdown; this doc is
the durable distillation so 012's plan does not need to re-read the
pack.

---

## Command identity

The command is **`/speckit.clarify`** in the upstream `github/spec-kit`
project. It is **mandatory** — positioned between `/speckit.specify`
and `/speckit.plan` as a required step in the spec-kit workflow.

## Input

`speckit.clarify` takes a `spec.md` file as input. It does not take a
feature directory, a free-form topic, or a list of questions. The
spec must already exist (written by `/speckit.specify` or equivalent).

## Interactive behavior

Clarify asks **up to 5 targeted clarifying questions** at a time. The
questions are generated by a taxonomy-driven coverage scan across 10
categories (see Scope below). The operator answers each question
interactively, and answers are recorded into the spec.

Questions are prioritized by impact: architectural significance, data
modeling impact, task decomposition, test design, UX behavior,
operational readiness, compliance validation. Stylistic preferences
and plan-level execution details are explicitly excluded.

## Output shape

Clarify modifies `spec.md` **in place** — it does not write a separate
artifact. The changes are:

### `## Clarifications` section (created if absent)

Added at a canonical position in the spec. Subsequent clarify runs
append new session subsections rather than overwriting.

### `### Session YYYY-MM-DD` subheader

One per clarify run. Session date is recorded in the subheader, but
no agent name, no tool version, no UUID, no signature.

### Q&A bullets

Inside each session subsection:

```markdown
### Session 2026-04-11

- Q: <question text> → A: <final answer>
- Q: <next question> → A: <next answer>
```

Note the `→` (arrow) separator between question and answer.

### Inline spec updates

As a clarify session resolves a question, it also **writes the
answer into the relevant section of the spec** — Functional
Requirements, Data Model, Success Criteria, Edge Cases, etc. These
are marked with minimal, testable insertions. The `## Clarifications`
section is the durable record of the Q&A; the inline updates are
the resolved content.

## Scope: the 10 taxonomy categories

Clarify's coverage scan explicitly addresses these areas. A downstream
`review-spec` command **must assume these are covered** and avoid
re-asking:

1. **Functional Scope & Behavior** — happy-path flows, core FRs
2. **Domain & Data Model** — entities, attributes, relationships
3. **Interaction & UX Flow** — user-facing sequences, error paths
4. **Non-Functional Quality Attributes** — performance, scalability,
   reliability, observability, security / privacy, compliance
5. **Integration & External Dependencies** — external systems, APIs,
   data sources, contracts
6. **Edge Cases & Failure Handling** — what if X, fallbacks, retries
7. **Constraints & Tradeoffs** — hard limits, platform assumptions,
   explicit non-goals
8. **Terminology & Consistency** — vocabulary alignment within the
   spec
9. **Completion Signals** — how the operator knows it's done
10. **Misc / Placeholders** — TBD markers, open questions the author
    left in the draft

For each category, clarify marks the scan as **Complete**, **Partial**,
or **Missing**. Partial and Missing categories generate clarification
questions.

## Scope boundaries — what clarify does NOT cover

This is the exact scope `review-spec` should own, because clarify is
**explicitly narrow** and these items are out of its scope by design:

1. **Cross-spec consistency** — clarify does not compare against
   other features or specs in the repo. It cannot see that FR-005 in
   this spec contradicts FR-012 in a sibling spec.
2. **Feasibility & implementation tradeoff analysis** — clarify may
   ask about constraints, but it does not audit whether the spec is
   actually implementable within those constraints, nor does it
   surface tradeoffs between alternative implementations.
3. **Security / compliance audit** — clarify asks about security and
   privacy requirements as clarifying questions (under
   non-functional quality attributes), but it does NOT audit threat
   models, attack surfaces, regulatory gaps, or data-handling
   compliance.
4. **Dependency graph analysis** — clarify lists external APIs
   mentioned in the spec but does not analyze version conflicts,
   breaking-change exposure, or upstream reliability.
5. **Comparison against industry patterns or alternatives** — clarify
   narrows scope (intentionally). It does not say "this is unusual"
   or "have you considered pattern X."

## Integration contract for `review-spec`

Based on the above, `review-spec` should:

### Precondition check

Before running, `review-spec` should verify clarify ran on the target
spec. The simplest check is:

```python
spec_text = Path(spec_path).read_text(encoding="utf-8")
if "## Clarifications" not in spec_text:
    raise ReviewSpecError(
        f"review-spec requires /speckit.clarify to have run first. "
        f"No '## Clarifications' section in {spec_path}. "
        f"Run `/speckit.clarify` and answer the questions, then retry."
    )
```

More robust check: verify at least one `### Session YYYY-MM-DD`
subheader exists inside `## Clarifications`. This guards against a
stub `## Clarifications` heading with no actual content.

### Scope enforcement

`review-spec` is **cross-only** per 012's brainstorm — no self-pass.
Its scope is the 5 things clarify does not cover:

1. Cross-spec consistency
2. Feasibility / tradeoff analysis
3. Security / compliance audit
4. Dependency graph analysis
5. Industry-pattern comparison

The command prompt for `commands/review-spec.md` (written later in
012's implementation phase) should explicitly reference this list and
instruct the adversarial agent to **not** re-ask questions already
covered by clarify's 10 taxonomy categories.

### Output shape

Per the 012 brainstorm:

```markdown
# Review: Spec

## Prerequisites
- Clarify session: YYYY-MM-DD (required — must exist in spec.md `## Clarifications`)

## Cross Pass (agent: <name>, date: YYYY-MM-DD)
- Cross-spec consistency findings
- Feasibility / tradeoff findings
- Security / compliance findings
- Dependency graph findings
- Industry-pattern comparison notes

## Verdict
- status: ready | needs-revision | blocked
- rationale: ...
- follow-ups: ...
```

The `## Prerequisites` block records which clarify session the review
was run against. If clarify runs again after review-spec, the review
should be re-run so the prerequisites stay aligned.

## Open integration question

**When clarify adds new sessions, does `review-spec` need to re-run?**

If clarify runs N times, the `## Clarifications` section has N
session subsections. The last review-spec pass only reviewed the spec
as of session M (where M <= N). If clarify added sessions after the
review, the review is stale.

**Recommendation for 012 plan**: `review-spec` records the
**latest clarify session date** in its Prerequisites block at review
time. On any subsequent clarify run, flow-state or matriarch should
surface the review as **stale** — the verdict is no longer current.
A simple date comparison handles this.

## Anti-scope

Things `review-spec` should explicitly NOT do, even though they might
seem in-scope:

- **Do not ask new clarifying questions.** If the reviewer thinks
  clarify missed something, the right response is to request a
  clarify re-run, not to ask the author directly.
- **Do not edit the spec.** `review-spec` is read-only against the
  spec. Any changes come from the author responding to the verdict
  and optionally re-running clarify.
- **Do not second-guess clarify's coverage.** If clarify says a
  category is Complete, trust it. The review's job is the 5 items
  clarify cannot reach, not to retread its 10 categories.

## Summary

| Concern | Owned by |
|---|---|
| Spec sharpening via Q&A | `speckit.clarify` (upstream) |
| Inline spec resolution of ambiguity | `speckit.clarify` (upstream) |
| Durable Q&A record | `## Clarifications` section in `spec.md` |
| Cross-spec consistency audit | `review-spec` (new) |
| Feasibility / tradeoff audit | `review-spec` (new) |
| Security / compliance audit | `review-spec` (new) |
| Dependency graph analysis | `review-spec` (new) |
| Industry-pattern comparison | `review-spec` (new) |
| Adversarial framing | `review-spec` (new, always cross-mode) |

This is a clean split. Upstream handles the author-facing sharpening
loop, Orca handles the adversarial review pass. Neither duplicates
the other.
